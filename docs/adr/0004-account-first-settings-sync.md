# ユーザー設定はアカウント優先で同期し、初回ログイン時のみローカルを引き継ぐ

ログインは任意（ゲストプレイが前提）のまま、ユーザー設定をアカウントに保存する。衝突解決は「アカウント優先」：ログイン時に DB へ保存済みの設定があれば常にそれが勝ち、端末（localStorage）の設定は上書きされる。唯一の例外は初回引き継ぎで、DB に設定が一度も保存されていない初回ログイン時だけ、ゲスト時代のローカル設定を DB へ取り込む。ログイン中の保存は DB と localStorage の二重書きとし、localStorage は常に「最後に適用された設定」のミラーになる（ログアウト後も見た目が巻き戻らない）。

**理由**: デバイスを跨いだとき「アカウントの設定がついてくる」というユーザーの期待に一致し、どちらが勝つかが常に一意に決まるため。ローカル優先はログインのたびに端末の古い設定が DB を上書きして同期の意味が消え、都度確認ダイアログは設定項目の軽さに対して UX コストが過剰。

## Considered Options

- **常にローカル優先** — 却下。別端末でログインするたびにその端末の古い設定が DB を上書きし、デバイス間同期が成立しない。
- **衝突時にユーザーへ選ばせる** — 却下。テーマや音量程度の重みに対して確認ダイアログは過剰。
- **ゲストにも匿名 ID で DB 行を発行** — 却下。デバイスを跨げないのにサーバーコストとゴミ行だけが増える。

## Consequences

- **サインアップ時に user_settings 行を自動作成してはならない。** 行を先に作ると「DB が空」という状態が消滅し、初回引き継ぎが永久に発火しなくなる。行は初回の保存（PUT /api/settings）で遅延作成し、shared props の `settings: null` が「未保存＝引き継ぎ対象」のクライアント側シグナルとなる。
- サーバー契約は設定画面に露出している項目のみ（アカウントと設定の用語は CONTEXT.md 参照）。UI にない設定フィールドを DB スキーマへ先取りしない。
