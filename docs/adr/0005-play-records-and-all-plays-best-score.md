# プレイ記録を新設し、ベストスコアを全プレイ基準に統一する

プレイ結果を残す経路が現状ひとつも無い（`scores` はランキング登録したプレイだけ、IndexedDB 経路はアプリ未接続）。プレイ履歴画面のために、ログインユーザーの完了プレイを1件ずつ自動保存する **プレイ記録**（`game_results` テーブル）を新設する。あわせて **ベストスコア** の母集団を「ランキング登録したスコア」から「全プレイ記録」へ再定義し、ヘッダーの共有プロップ（`User#best_scores`）と履歴サマリーが同一導出（`game_results` 由来）を共有するようにする。ランキング登録は従来どおり別個の任意操作として `scores` に残る。

**理由**: 「ベストスコア＝自己最高記録」がユーザーにとって最も自然で、自己ベストの定義が「公開したか否か」に引きずられるのは不自然。ヘッダーと履歴で同じ語が別の値を指す事故（登録し忘れたプレイのほうが高い、等）を、両者を同一メソッドから導くことで構造的に防ぐ。プレイ記録は最終スコアを丸ごと（正解札数・正確率・WPM・最大コンボ＋モード/難易度/スコア or タイム/プレイ日時）保存する ── この issue が存在する理由そのもの（記録しなかった値は永久に戻らない）に従い、終了画面に出ている瞬間に全部残す。

## Considered Options

- **(却下) ベストスコアはランキング登録基準のまま、履歴サマリーだけ全プレイ** — 同じ語で母集団が2つに割れ、履歴テーブル（全プレイ）とサマリー（登録のみ）が食い違って見える。
- **(却下) 別語で共存** — ヘッダー＝登録基準の「ベストスコア」、履歴＝全プレイ最高を別語に。CONTEXT.md の Avoid（自己ベスト/ハイスコア/最高記録）を踏まない命名が難しく、1語1意味の原則からも遠い。
- **(却下) バックフィル（B2）** — 既存 `scores` を `game_results` へ種蒔きしてベストスコアを保つ案。だが過去 `scores` には `accuracy`/`max_combo` が無く、過去の未登録プレイは元々どこにも無いため、履歴が「登録分だけ」歯抜けになりノイズが増える。

## Consequences

- **切替時、過去の登録ベストはヘッダーから一旦消える（クリーンカット B1）**。`game_results` は新テーブルで過去プレイを持たず、過去 `scores` からの完全復元も不可（`accuracy`/`max_combo` 欠落）。ベストスコアはプレイ記録導入後のプレイだけを母集団に再スタートする。開発中でこの後退が許容できることを前提に採用した。
- **`User#best_scores` の導出元を `scores`→`game_results` へ作り替える。** 「画面実装」に見えて、実体は記録永続化とベストスコア導出の作り替えを含む。ヘッダー共有プロップと履歴サマリーは必ず同一メソッドを通す。
- **プレイ記録はログインユーザーのみ・自然完了時のみ。** ゲストは発行しない（ADR 0004 の方針と一貫）。中断・離脱・リロードは残さないため、ランキング登録できるプレイ（完了時のみ可）と一致し、`scores` ⊆ プレイ記録 が成り立つ。
- **`score`/`time_ms` はモード依存で保持**（ランダム＝`score`、タイムアタック＝`time_ms`、他方は NULL）。`scores` と同じ PG enum（`game_mode`/`difficulty`）を再利用する。
